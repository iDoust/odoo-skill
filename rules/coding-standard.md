# Odoo Coding Standards

These rules enforce consistency, maintainability, and compatibility with the Odoo Community Association (OCA) guidelines. All AI agents must adhere to these standards when writing or modifying Odoo code.

## 1. General Principles

- **Language:** All code, variables, and docstrings MUST be in English.
- **Python Conventions:** Follow PEP8 strictly.
- **SOLID & DRY:** Do not repeat code. Use inheritance (`_inherit`) and composition effectively.
- **KISS:** Keep methods focused. If a method exceeds 50 lines, it probably needs to be split.
- **`super()`:** Use `super()` without arguments (Python 3 style).

## 2. File and Directory Structure

- Python models go in `models/`
- XML views go in `views/`
- Data files (master data, cron, sequences) go in `data/`
- Security definitions go in `security/`
- Static assets (CSS, JS, XML templates) go in `static/src/`
- **Imports:** The `__init__.py` files must only import the necessary subdirectories or files.

## 3. Python Import Ordering

All imports must be at the top of the file, ordered and separated by a blank line as follows:

1. Standard library imports (e.g., `import logging`, `import datetime`)
2. Third-party library imports (e.g., `import requests`)
3. Odoo core imports (e.g., `from odoo import api, fields, models, _`)
4. Odoo exception imports (e.g., `from odoo.exceptions import UserError`)

**Never import inside functions or methods.**

## 4. Naming Conventions

### Models and Tables
- **Model Name (`_name`):** Dot-separated, singular. (e.g., `sale.order`, `school.student`)
- **Python Class Name:** CamelCase. (e.g., `SaleOrder`, `SchoolStudent`)
- **Table Name:** Automatically generated by replacing dots with underscores (`sale_order`). Do not define `_table` unless bypassing this rule for a specific legacy reason.

### Fields
- **Many2one:** Must end with `_id` (e.g., `partner_id`, `company_id`).
- **One2many / Many2many:** Must end with `_ids` (e.g., `line_ids`, `tag_ids`).
- **Date/Datetime:** Should clearly indicate time (e.g., `date_order`, `create_date`).
- **Boolean:** Should start with `is_` or `has_` (e.g., `is_company`, `has_active_contract`).

### Methods
- **Compute Methods:** Must start with `_compute_` (e.g., `_compute_total_amount`).
- **Inverse Methods:** Must start with `_inverse_` (e.g., `_inverse_total_amount`).
- **Search Methods:** Must start with `_search_` (e.g., `_search_total_amount`).
- **Onchange Methods:** Must start with `_onchange_` (e.g., `_onchange_partner_id`).
- **Constraint Methods:** Must start with `_check_` (e.g., `_check_date_end`).
- **Action Methods:** Methods bound to UI buttons should start with `action_` (e.g., `action_confirm`, `action_cancel`).

### Variables
- Use descriptive names that reveal intent.
- **Forbidden generic names:** `data`, `info`, `temp`, `tmp`, `var`, `obj`, `item`, `val`, `recs`, `res`.
- **Allowed:** `order_values`, `partner_record`, `product_id`.
- Use `record` or specific singular names for loop iteration: `for order in self:` (not `for x in self:`).

## 5. XML ID Naming Conventions

- **Form View:** `model_name_view_form` (e.g., `sale_order_view_form`)
- **List View:** `model_name_view_tree` (e.g., `sale_order_view_tree`)
- **Search View:** `model_name_view_search` (e.g., `sale_order_view_search`)
- **Action:** `model_name_action` (e.g., `sale_order_action`)
- **Menu:** `model_name_menu` (e.g., `sale_order_menu`)

## 6. CSS / SCSS — Scoped to Module ⚠️

NEVER write global CSS that can bleed into Odoo's core UI or other modules.
All CSS/SCSS rules MUST be scoped under a unique class prefixed with the module name.

```scss
// ❌ BAD: Global selector — affects all Odoo forms
.o_form_view {
    background: red;
}

// ✅ GOOD: Scoped to this module only
.my_module_form {
    .o_field_widget {
        color: #714b67;
    }
}
```

In the form view XML, apply the wrapper class at the top level:
```xml
<!-- ✅ Wrap your view with a module-specific class -->
<form class="my_module_form">
    ...
</form>
```

For SCSS files, place them in `static/src/scss/my_module.scss` and register in `__manifest__.py` only inside the correct asset bundle:
```python
'assets': {
    'web.assets_backend': [
        'my_module/static/src/scss/my_module.scss',
    ],
},
```

## 7. Anti-Over-Engineering ❌

Custom Odoo modules should be as **simple as possible**. Complexity adds maintenance cost and breaks on upgrades.

- **Don't create new models** when inheriting an existing model (`_inherit`) is sufficient.
- **Don't create a wizard** if the action can be done with a simple button on the form.
- **Don't add a configuration setting** unless the client explicitly needs to toggle the behavior.
- **Don't create computed stored fields** when a simple method call would do.
- **Don't override `create()` and `write()`** unless there is no alternative (prefer `@api.depends` and `@api.constrains`).
- **Don't add new menus or actions** if the feature can live inside an existing menu item.
- **Ask yourself first**: Can I solve this with **Odoo Studio**, a **server action**, or a **scheduled action** instead of Python code?

## 8. Other Anti-Patterns to Avoid ❌

- **No `# -*- coding: utf-8 -*-`**: It is redundant in Python 3.
- **No `api.multi` or `api.one`**: These are obsolete in modern Odoo.
- **No Hardcoded IDs**: Never use `browse(42)`. Use `env.ref('module.xml_id')`.
- **No Raw SQL**: Use ORM whenever possible. If raw SQL is needed for performance, use Odoo's `SQL()` builder (v18+).

## 7. Source Referencing

When implementing Odoo-specific behavior, **ALWAYS** document the Odoo source code location that serves as the reference.

```python
# @see https://github.com/odoo/odoo/blob/17.0/odoo/fields.py
```
